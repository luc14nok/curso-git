Clase 1 GIT

Control de versiones

Areas de git

Working directory

es la carpeta en la que iniciamos el repositorio, donde realizamos los cambios en los archivos y muy rara vez estaremos utilizando la consola.

  • dir / ls comandos que permiten mostrar el contenido del directorio actual dependiendo del CLI podemos usar uno o el otro
  • mkdir <directorio crea un directorio con el nombre especificado. Debemos tener en cuenta, en caso que este lleve espacios deberá estar entre comillas o separado con contrabarras, por ejemplo mkdir "proyecto git" o proyecto\git
  • cd <directorio cambia el directorio actual por la ruta especificada, siempre y cuando la misma exista. Para utilizarlo simplemente debemos escribir algo como cd proyecto\git
  • rm <file Elimina un archivo o directorio especificado, si queremos eliminar un repositorio, podemos escribir rm -rf.git dentro de la carpeta donde se haya inicializado el mismo
  • git init inicia o reinicia un repositorio de git para comenzar a registrar los cambios de un proyecto

Staging area (o index)

Es el area de control de cambios. Es un archivo que identifica y verifica los cambios realizados en el directorio de trabajo. Todo archivo al que querramos hacerle seguimiento debe ser agregado previamente al INDEX para que se realice su snapshot.

  • git status compara el working directory con el archivo index y notifica sobre las diferencias que existen entre ambos. Este comando notifica sobre el estado de los archivos
    • Untracked: el archivo no fue agregado al index.
    • Tracked: archivo al que se está haciendo seguimiento
    • Modified: archivo que difiere con respecto a lo registrado en el index
    • Commited: archivo cuyos cambios ya fueron confirmados
  • git add <file> agrega al area de control de cambios (INDEX) los archivos seleccionados, podemos acelerar esta tarea utilizando la expresion git add . , pero debemos tener en cuenta que esto agregará todos los cambios del directorio actual
  • git rm --cached <file> remueve el archivo seleccionado del INDEX pero mantiene los cambios en el directorio de trabajo (WORKING DIRECTORY). Esto funciona en el caso que hayamos agregado el archivo al index y todavia no exista en el historial.
  • git restore <file> reestablece el estado del archivo seleccionado a como se encontraba en el index antes de realizar los cambios.

Repository (local o remoto)

Es el area donde se almacenan los cambios confirmados en formato blob (binary object file). esto nos permite reestablecer, tanto archivos, directorios o hasta el mismo proyecto utilizando como referencia dichos archivos e indicando a GIT en que instancia del mismo

  • git diff <referencia> compara el working directory con el commit de la referencia especificada. Podemos tambien indicar mediante parametros algun archivo en particular como por ejemplo git diff HEAD^ -- index.html .
    • refrencias absolutas es un número de hash (único) que se especifica en el commit dentro del historial del cambios
    • referencias relativas se refiere a cuantos ancestros (head^) o commits anteriores (HEAD~2) se encuentra nuestro punto de comparación.
  • git blame <file> muestra el numero de commit, el autor, la fecha y hora en la que se realizaron los cambios en una carpeta o archivo. Incluso muestra cuando hay cambios pendientes por confirmar
  • git commit abre el editor para realizar la confirmacion de cambios y generar el commit pertinente con mensaje y datos del autor. Una forma rápida de utilizar este comando para generar el mensaje es git commit -m "mensaje" o tambien podemos utilizar git commit -am "mensaje" si queremos agregar (add) los cambios de los archivos a los que les estemos haciendo seguimiento y confirmarlos con un solo comando
  • git checkout <referencia> permite navegar dentro del historial de commits utilzando referencias relativas, absolutas y apuntadores de cualquier tipo, especialmente si queremos volver al ultimo commit emitido
